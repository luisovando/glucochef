# ADR-004 — Hide the LLM behind an internal port so OpenAI and Claude are interchangeable

**Date:** 2026-05-25
**Status:** Accepted
**Deciders:** Luis Ovando

## Context
Ingredient alternatives and recipe text are generated by a large language model. Both OpenAI and Claude are viable for the MVP; pricing, rate limits, and academic-grade availability can shift between now and the July deadline. Coupling `Suggestion Service` and `Recipe Service` directly to a single SDK would make the system fragile against a provider outage, a sudden quota change, or a final cost decision. The product also has a strict rule that no raw PHI leaves the trust boundary, which is easier to enforce in a single shared adapter than in two separate call sites.

## Decision
`Suggestion Service` and `Recipe Service` depend on an internal `AIProvider` port — a small Python protocol with `complete(prompt, schema)` semantics — and not on any vendor SDK. Two adapters are implemented behind the port (OpenAI and Claude); the active adapter is selected by configuration. The port is the single place that constructs prompts; it is also the single place that strips identifiers and raw lab values from any payload before the call.

## Consequences
- **Positive:** Switching providers is a configuration change and a focused contract test, not an architectural change. PHI scrubbing has exactly one home, which is straightforward to audit. Prompts and outputs can be logged uniformly into `memory-bank/ai-usage-notes.md` regardless of provider, satisfying the AI usage log deliverable.
- **Trade-off:** A small abstraction layer is added even though only one provider is active at a time in production. The cost is bounded (one Python protocol, two adapters) and pays for itself the first time the provider has to be swapped.
- **Risks mitigated:** Avoids vendor lock-in within the academic timeline, isolates the team from rate-limit surprises, and centralises the de-identification rule that protects the patient.
